Jira Ticket Briefing Skill

Problem

When picking up an existing Jira ticket (especially one with an open PR), you need fast answers to:





What is broken, for whom, and how to reproduce it?



Where in a large codebase should I start reading/debugging?



If a PR already exists, what exactly does it change and does it fully address the ticket?

Solution shape

A personal Cursor skill you invoke in chat with a Jira key/URL or a PR URL directly (GitHub or Bitbucket). The agent runs a fixed pipeline: fetch ticket and/or PR data, load your codebase map, search the open workspace, then output a structured briefing.

flowchart TD
    userInput["User: Jira key/URL OR PR URL"] --> skill["jira-ticket-brief skill"]
    skill --> route{Input type?}
    route -->|Jira| fetchJira["fetch-jira script"]
    route -->|PR| fetchPR["fetch-pr script"]
    fetchPR --> maybeJira{Jira key in PR?}
    maybeJira -->|yes + creds| fetchJira
    fetchJira --> hasPR{Linked PR found?}
    hasPR -->|yes| fetchPR
    skill --> loadMap["Read CODEBASE.md if present"]
    fetchJira --> searchRepo["Repo search: grep + read"]
    fetchPR --> searchRepo
    loadMap --> searchRepo
    searchRepo --> briefing["Structured briefing markdown"]
    briefing --> cursorChat["Continue debugging in same chat"]

Defaults: Jira Cloud REST API + GitHub (gh CLI) + Bitbucket REST API.

Three briefing modes (automatic)

The skill detects mode from input and fetched data — no separate command needed.

Mode A — Ticket-only (no linked PR)

Use when you are starting fresh: writing tests, reproducing, or investigating before a fix exists.

Briefing sections:





Ticket summary — title, status, priority, assignee, labels



Problem statement — distilled from description + comments (what broke, impact, regression vs new)



Reproduction — extracted steps, env/preconditions, known workarounds from comments



Acceptance criteria — from description, custom Jira field, or linked test cases



Codebase entry points — ranked list of files/dirs to open first, with rationale



Search evidence — grep hits for error strings, API paths, class names from the ticket



Suggested investigation plan — ordered checklist (repro → read X → add logging → write test)



Open questions — ambiguities the ticket does not resolve

Mode B — Ticket + PR analysis

Use when a PR is already linked (Jira development panel, remote link, or PR title/body mentions the ticket key).

Everything in Mode A, plus:
9. PR overview — link, author, state, base branch, commit summary
10. Fix analysis — what root cause the PR targets, approach taken, files touched grouped by concern
11. Diff walkthrough — per changed file: what changed and why it matters
12. Coverage check — does the PR address all acceptance criteria? edge cases mentioned in comments?
13. Review/testing focus — what to manually test, which existing tests to run, suggested new tests
14. Risks / gaps — missing tests, unrelated changes, areas the PR does not touch

Mode C — PR-only (direct PR input)

Use when you paste a GitHub or Bitbucket PR URL directly. If a Jira key appears in the PR and credentials are set, Mode B is used instead.





PR overview — title, URL, author, state, branches



Intent — what the PR accomplishes from description and diff



Detected Jira keys — keys found in title/body (note if Jira was not fetched)



Change summary — files touched, per-file diff stats



Codebase entry points — ranked files from diff + CODEBASE.md + grep



Risks and test plan



Suggested follow-ups



Data gathering design

Jira (fetch-jira.mjs)

Call Jira Cloud REST API v3:





GET /rest/api/3/issue/{key}?expand=renderedFields,names



Fields: summary, description, status, priority, comment, labels, issuelinks



Remote links / dev status for PR URLs (parse GitHub PR links from remotelink or issue properties)



Optional config for custom fields (common ones: repro steps, acceptance criteria, environment)

Output: JSON file the agent reads (keeps skill instructions small and reproducible).

GitHub PR (fetch-pr.mjs)

Use gh (already in your workflow):





gh pr view <url|number> --json title,body,state,author,baseRefName,headRefName,commits,files,reviews



gh pr diff <url|number> --name-only for file list



gh pr diff <url|number> capped/truncated for large diffs (script summarizes per-file hunks rather than dumping 10k lines into context)

If multiple PRs link to the ticket, fetch all and note which is primary (most recent open, or explicitly linked in Jira).

Codebase map (CODEBASE.md)

A per-repo markdown file you maintain (does not need to be exhaustive). Suggested structure:

# Codebase Map

## Domains

### Billing
- Paths: `src/billing/`, `packages/billing-api/`
- Key entrypoints: `BillingService.ts`, `routes/billing.ts`
- Tests: `tests/billing/`
- Related Jira components: Billing, Invoicing

### Auth
- Paths: `src/auth/`
- ...

The skill reads this first for coarse routing, then validates/refines with live repo search.

Dynamic repo search (agent-native)

After scripts return structured data, the skill instructs the agent to:





Extract search terms from ticket text: error messages, endpoint paths, UI labels, stack traces, class/function names



Cross-reference terms against CODEBASE.md domain keywords



Run targeted grep / file search in the open workspace



Rank candidates: direct string match > domain map match > import/call graph proximity (when obvious from filenames)

This hybrid (static map + dynamic search) is what makes it work on massive codebases without maintaining a full index.



Skill invocation UX

Trigger phrases (in skill description): "brief ticket", "jira context", "analyze PROJ-123", paste Jira URL, paste GitHub/Bitbucket PR URL.

Typical sessions:

You: Brief PROJ-456
Agent: [runs brief.mjs, searches repo, outputs Mode A or B briefing]

You: Analyze https://github.com/org/repo/pull/8821
Agent: [runs brief.mjs, fetches PR, optionally Jira if key detected, outputs Mode B or C]

You: What's in PR 42 for org/repo?
Agent: node scripts/brief.mjs 42 --repo org/repo

The briefing becomes durable context for the rest of the chat — you do not need to re-explain the ticket.

Implementation phases

Phase 1 — MVP (ticket-only briefing)





Create skill + fetch-jira.mjs



Hard-coded output template for Mode A



Agent does grep-based repo search against open workspace



Done when: pasting PROJ-123 in a repo produces a usable "where to start" briefing without a PR

Phase 2 — Codebase map integration





Add CODEBASE.md template to skill reference docs



Skill auto-discovers map file from config paths



Briefing ranks entry points using map + grep evidence together

Phase 3 — PR analysis mode





Add fetch-pr.mjs + PR link detection from Jira



Extend template with Mode B sections (fix analysis, coverage check, risks)



Handle large diffs via per-file summaries

Phase 4 — Quality-of-life (optional, later)





Cache ticket JSON locally (avoid re-fetching same ticket in one session)



Post briefing summary back as Jira comment (opt-in)



Support Jira Server/DC or GitLab if your org uses them



Example briefing output (abbreviated)

# Briefing: PROJ-456 — Checkout fails for EU VAT customers

## Problem
Payment intent creation fails when `billingAddress.country` is in the EU VAT list...

## Reproduction
1. Log in as merchant on staging
2. Create order with DE billing address
3. Observe 422 from `/api/v2/payments/intents`

## Start here (ranked)
1. `src/payments/VatValidator.ts` — direct match on error message in ticket
2. `src/payments/routes/intents.ts` — API route from repro steps
3. `tests/payments/vat.test.ts` — existing coverage, likely place to add repro

## PR analysis (PR #8821 — open)
**Fix approach:** Adds missing VAT ID normalization before validator call.
**Touches:** 4 files across payments + shared geo utils.
**Gap:** No test for FR addresses; comment from @alex mentions this edge case.

## Suggested next steps
- [ ] Reproduce on staging with steps above
- [ ] Read VatValidator + intents route
- [ ] Add failing test for DE case before reviewing PR diff



Risks and mitigations







Risk



Mitigation





Jira tickets are vague



"Open questions" section; agent asks 1–2 clarifying questions only when blocking





No PR link in Jira



Fall back to gh pr list --search "PROJ-123" as secondary lookup





Huge PR diffs blow context



Script summarizes per-file; agent reads full files only for top N changed files





Codebase map stale



Dynamic grep always runs; map is a hint, not source of truth





Wrong repo open in Cursor



Skill checks ticket components/labels against map; warns if no grep hits



Prerequisites before building





Jira API token — Atlassian account settings → Security → API tokens



**gh authenticated** — gh auth status should succeed



One pilot repo with a docs/CODEBASE.md stub (even 20 lines helps)



One real ticket (with and without a PR) to validate output quality



Success criteria





From Jira key to actionable briefing in under ~2 minutes



Entry-point file list is accurate enough that you open the right area on first try ≥80% of the time (tune with your map + grep rules)



PR mode clearly explains what was implemented vs what the ticket asked for



Briefing is reusable as chat context for writing repro tests or reviewing someone else's fix
